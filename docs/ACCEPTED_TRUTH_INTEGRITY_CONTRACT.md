# Accepted truth and project-tag integrity contract

Status: implementation proof in progress on `fix/accepted-truth-integrity`.
Date: 2026-07-22

This contract defines the combined A-06, A-07, and A-10 boundary. Accepted
Project Capsule truth, supplemental project metadata, and exact project-tag
assignments must not bypass their own revision, verification, or concurrency
rules.

## Accepted truth and supplemental metadata

`ProjectCapsuleTruthService.acceptPatch` accepts only the canonical Capsule
truth fields. Unknown keys and non-truth keys fail before any project, ledger,
or audit write. `lessonsLearned` is intentionally outside the truth hash and is
written only through the narrow supplemental metadata service.

AppState and local identity enrichment split mixed requests into canonical
truth and supplemental fields inside one database transaction. Supplemental-
only edits still validate the caller's expected truth revision. A failed truth
validation, stale revision, supplemental write, derived-tag write, or audit
step rolls the whole mixed workflow back.

## Verified source evidence

Source-based replay or recovery evidence is selected only after the complete
project revision ledger passes content-hash, contiguous-number, parent-link,
and exact-diff verification. Corruption anywhere in the chain fails closed,
including when the corrupt revision does not match the requested source.

Status proposals retain verified truth-source recovery. A partially applied
manifest proposal cannot be recovered from truth evidence alone because that
evidence proves neither supplemental metadata nor the resulting project-tag
set. Without an approved composite transaction receipt, both approval and
rejection fail with `proposal_unverifiable_manifest_replay` and require manual
repair/recreation.

## Manifest proposal snapshot

The server captures `atlas.manifest_proposal_base.v1` inside one database read
transaction. The snapshot contains the project ID, fully verified truth
revision ID, and a SHA-256 digest of the exact assigned project tags. The tag
projection is sorted by stable tag ID and contains `{id, normalized name,
color}`. Raw assignment IDs are resolved exactly; dangling assignments make
the snapshot invalid instead of disappearing through an inner join.

After the draft review claim and before any truth, supplemental, tag-creation,
or assignment write, approval revalidates the project, truth revision, and tag
digest. Metadata-only and tags-only proposals validate both halves. Missing or
malformed legacy snapshots, deleted projects, stale truth, stale tags,
dangling assignments, and ambiguous case-insensitive tag names fail with
stable typed conflicts while the outer transaction restores the pending draft.

Manifest `tags` must be absent or an array of unique, nonblank strings. Absent
preserves assignments; present-empty clears them. Tag lookup/creation occurs
only after snapshot validation and rejects ambiguous case variants before
creating any row.

## Atomic write and replay boundary

Truth revisions, supplemental metadata, tag creation and assignment, review
state, and audit events commit in the existing proposal approval transaction.
Crashes at any write hook roll all of them back. Approved-draft replay returns
the stored terminal result without repeating side effects. The MCP proposal
tool never accepts a client-supplied base snapshot; it stores the server-owned
snapshot generated by the service.

Focused proof covers strict field routing, stale supplemental edits, corrupt
ledger ancestors, snapshot shape and ownership, stale truth/tag cross-races,
assignment/name/color changes, absent versus empty tags, legacy inputs,
dangling assignments, case ambiguity, deleted projects, crash rollback,
approved replay, MCP creation, enrichment, and two-connection same-base
contention. A-08 full-history verification performance and A-09 task-tag clear
semantics remain separate findings.
