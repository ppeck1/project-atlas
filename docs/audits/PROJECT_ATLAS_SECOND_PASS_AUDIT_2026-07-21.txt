Project Atlas main — second-pass audit
Verdict

The immediate Capsule goal was substantially accomplished. The larger human/LLM operating loop was not.

Capsule is now a real accepted-truth subsystem rather than a conceptual dashboard:

human-authored project truth is editable;
changes receive field-level review;
acceptance is explicit;
revisions are immutable;
stale saves are rejected;
agent-originated changes remain proposals.

That design is present in the current README, architecture, database, services, migrations, tests, and UI.

But the product plan correctly says that Workboard orchestration, configurable loops, one closed Codex workflow, compact MCP access, and usefulness measurement remain future phases.

The strongest part is the Capsule truth model. The weakest parts are:

destructive recovery failure handling;
concurrent LLM queue operation;
proposal application idempotency;
Workboard’s fit with the Capsule-centered product;
formal acceptance and documentation closure.

I found 49 actionable findings below. Several are failure-path defects rather than bugs a single user will hit during ordinary use. They still matter because recovery and agent orchestration are specifically supposed to provide trustworthy boundaries.

Audit basis and limitations

I reviewed the uploaded main bundle and the current public GitHub state through the recent Capsule, recovery, project search, reactive data, and public-documentation merges.

I examined the high-risk paths covering:

Capsule truth and revision history;
migrations and persistence;
full backup, staging restore, and live replacement;
LLM queue leasing and completion;
proposal creation, approval, and application;
workload ranking and recommendations;
runtime command execution;
local and remote MCP boundaries;
secret storage;
Workboard, Capsule, Projects, Operations, and Library UX;
CI, tests, public docs, and screenshots.

Locally verified:

Python policy/maintenance suite: 30/30 passed.
Missing relative Markdown links: 0.
Obvious private-key/API-key patterns in the public tree: 0 found.
Public PNGs contained only ordinary color/gamma/DPI metadata.
The Python remote MCP policy suite is one of the strongest parts of the repository.

I could not independently rerun Flutter analysis, Flutter tests, or the Windows release build because this environment lacks the Flutter/Dart/Windows toolchain. The merged PRs report up to 440 tests passed, 1 skipped, and successful Windows release builds.

The uploaded ZIP also has no .git directory, so its exact commit identity cannot be independently matched. Its contents appear consistent with current main.

Recovery assessment

The recovery design has the correct components—SQLite online backup, manifests, hashes, staging, typed confirmation, safety backup, and rollback. PR #25 reported a successful disposable end-to-end acceptance test.

The failure handling is not yet strong enough to support the phrase validated live recovery without qualification. R-01 and R-02 are the highest-priority defects in the repository.

Agent-loop assessment

The current queue and proposal path is suitable for a single operator, one worker, no-crash happy path.

It is not yet safe for:

multiple simultaneous workers;
retries after partial failure;
unattended agent completion;
stale task proposals;
strong exactly-once mutation claims.

That distinction is important. The product thesis is not wrong; the implementation needs transactional closure.

What is working well
1. The accepted-truth model is coherent

The implementation has one mutable accepted source—the project columns—and a separate immutable history. That avoids two competing mutable truths. Revisions include:

project-scoped sequence;
parent;
canonical content hash;
truth JSON;
field diff;
actor/source/reason;
acceptance timestamp.

Schema 25 adds update/delete prevention, while reads validate the complete chain. That is a meaningful implementation of the acceptance thesis, not merely terminology.

2. Capsule has a clearer product role

The current product sentence—resuming projects, choosing the next justified action, and collaborating with review-gated AI—is substantially better than presenting Atlas as an omnibus project dashboard.

The Act / Understand / Audit decomposition is also appropriate:

Act reduces the immediate decision surface.
Understand provides the contract and project context.
Audit exposes provenance, freshness, warnings, and verification.
3. The remote disclosure boundary is serious

The remote MCP path is narrow, read-only, allowlisted, deny-by-default, and tested against hidden projects, malformed output, response caps, unsafe policy rows, and semantic field leakage. That is stronger than the average local-first AI tool.

4. Public repository discipline is good

The current repository has:

real application screenshots;
an isolated public-safe fixture;
no broken relative documentation links;
no obvious public credential pattern;
deterministic verification commands;
explicit limitations;
a current Windows release path;
privacy-aware handoff material.
5. The recent refactoring strategy is correct

Incremental extraction from Project Detail and AppState is safer than a large architectural rewrite. The recent work on reactive people/risks/decisions, search, theme tokens, and separated services improves the product without replacing everything at once.

Final assessment

Project Atlas has crossed an important threshold: the central thesis now exists in code.

The Capsule goal was accomplished in the meaningful technical sense:

accepted truth is distinct from generated or derived state;
history is immutable;
stale acceptance is blocked;
human review is explicit;
agent changes remain proposals.

It has not yet accomplished the complete product goal because:

Workboard is not yet the shared agency surface;
the agent loop is not transactionally safe under concurrency or failure;
recovery contains critical rollback holes;
usefulness has not been measured;
formal Phase 1 acceptance remains undocumented.

The correct characterization is:

Capsule Phase 0/1 is implemented and conceptually successful. Project Atlas as a complete human/LLM operating system is still in the integration and product-validation stage.

The next work should be integrity hardening first, Workboard agency second, and measurement third.
